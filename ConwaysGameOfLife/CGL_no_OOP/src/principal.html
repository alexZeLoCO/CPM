What is Conway's Game of Life?

Conway's Game of Life is an Automaton, capable of creating and terminating cells in a grid, depending on a number of conditions.
It is described as a zero-player game, as the 'player' only controls the starting state of the grid.
Any given cell in the grid can have one of two states: Alive (represented by a 1) or dead (represented by a 0).
Before staring, the user must mark which cells are alive, once started, these cells will survive, die and come back to life depending on the number of alive neighbors it has.
This automata does not seem to have much capabilities, however it is Turing complete , so it is capable of simulating a Turing Machine, and any code from any programming language given the right inputs.

Who created this automaton?

Conway's Game of Life was invented by the mathematician John Horton Conway (26 December 1937, Liverpool, England - 11 April 2020, New Jersey, United States).
Some of John Horton Conway's areas of research are:
  • Number Theory
  • Combinatorial Game Theory
  • Algorithmics
  • Geometry

Templates and Designs

There are plenty of patters that are easily produced during the execution of any simulation. In a similar manner, many starting designs have been studied and named.
These can be organized into types depending on their 'movement'.
  1. Still lifes (or stable). These do not move. More examples
    • Block
    • Bee-hive
    • Loaf
    • Boat
    • Tub
  2. Oscillators. These switch between a finite number of positions ending in the starting position and repeating the process until interrupted. They have a peridod which is the number of cycles it takes for them to complete the pattern. More examples
    • Blinker - Period 2
    • Toad - Period 2
    • Beacon - Period 2
    • Pulsar - Period 3
    • Penta-decathlon - Period 15
  3. Spaceships. These move, across the matrix until they are either interrupted by other piece or out of bounds. More examples
    • Glider
    • Light-weight spaceship
    • Middle-weight spaceship
    • Heavy-weight spaceship
Even though this is one of the most important organization, there are others that are also frequently used to refer to certain behaviours:
  1. Methuselahs. These have an apparent erratic behaviour and do not follow any kind of loop or direction. However, it is most likely that after a given number of iterations, they will become stable.
    • R-pentomino - 1103 iterations before becoming stable
    • Diehard - 130 iterations before becoming stable
    • Acorn - 5206 iterations before becoming stable
  2. Guns. These create spaceships that are shot across the matrix
    • Gosper Glider Gun
    • Simkin Glider Gun

What are the rules?

There are 4 simple rules. These are applied on every cycle to determine the state of the next iteration.
  1. Any live cell with fewer than two live neighbours dies, as if by underpopulation.
  2. Any live cell with two or three live neighbours lives on to the next generation.
  3. Any live cell with more than three live neighbours dies, as if by overpopulation.
  4. Any dead cell with exactly three live neighbours becomes a live cell, as if by reproduction.
These rules can be simplified which will ease the implementation.
  1. Any live cell with two or three live neighbours survives.
  2. Any dead cell with three live neighbours becomes a live cell.
  3. All other live cells die in the next generation. Similarly, all other dead cells stay dead.

Gimmicks

These simple rules that may seem easy to understand and implement, have one caveat that can be overlooked at first glance.
If we make use of a matrix made of integers to emulate the grid, where 1 represents a live cell and 0 a dead cell, we must make an algorithm that evaluates every element of the matrix and switches 1s to 0s and viceversa when the right conditions are met.
For instance, this could be our matrix:
0 1 1
1 1 0
We start updating all the cells, starting with the one on the top left (matrix[0][0]).
To do this, we calculate the number of live neighbors of matrix[0][0] (3).
Given that matrix[0][0] is dead and has 3 neighbors, it should become a live cell. So we update its state:
1 1 1
1 1 0
We continue updating the next cell, the one on the top right (matrix[0][1]).
To do this, we calculate the number of live neighbors of matrix[0][1] (4).
Given that matrix[0][1] is alive and has 4 neighbors, it should die. So we update its state.
1 0 1
1 1 0
At this point, we have already made a mistake, beacuse matrix[0][1] should actually be alive.
The reason for this mistake is that we have been evaluating the same matrix we have been updating, so after the first substitution, we are not evaluating the original matrix anymore.
This is not the way the automaton is intended to play out, so we will need a secondary matrix. We will either copy the original matrix on to the secondary and use the secondary to evaluate as we modify the original or the other way around.
It must be taken into account however, that if we update the secondary matrix, we must update the original matrix once all cells have been evaluated.

Algorithm

These are some of the functions used to manage the logic of the game.
function neighbors (matrix, row, col) {
let count = 0; // int
if (row > 0) { // There is a cell on top
count+=matrix[row-1][col];
if (col > 0) { // There is a cell on the top left
count+=matrix[row-1][col-1];
}
if (col < matrix[row-1].length - 1) { // There is a cell on the top right
count+=matrix[row-1][col+1];
}
}
if (row < matrix.length - 1) { // There is a cell below
count+=matrix[row+1][col];
if (col > 0) { // There is a cell below left
count+=matrix[row+1][col-1];
}
if (col < matrix[row+1].length - 1) { // There is a cell below right
count+=matrix[row+1][col+1];
}
}
if (col > 0) { // There is a cell to the left
count+=matrix[row][col-1];
}
if (col < matrix.length - 1) { // There is a cell to the right
count+=matrix[row][col+1];
}
return count;
}
